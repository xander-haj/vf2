Author canonical content, test it through production systems, and safely generate game code.

Start here: make and save one change

  1. Choose Build, World, Creatures, or Logic, then pick something from the left library.
  2. A cube icon means the item always has a picture. Page and logic icons are settings only.
  3. Edit its fields in the right Inspector. Changes immediately update the center preview.
  4. Open Unsaved changes at the bottom to review exactly what changed.
  5. Resolve every item marked as an error in Checks.
  6. Select Save changes to validate the full snapshot and regenerate typed code.

Unsaved edits live only in memory. The game source changes only after a successful, revision-checked save transaction.

Understand the layout

Workspace tabs
Filter the project into Build, World, Creatures, or Logic without removing any content.
Left browser
Groups friendly names into visual objects and rules or data. Selecting an entry never edits it.
Center viewport
Uses production textures, meshing, generation, models, animation, and simulation code.
Right Inspector
Shows large visual tools first. All raw fields remain inside the Advanced details drawer.
Bottom panel
Shows whole-project compiler diagnostics or a field-level review of unsaved changes.

Build a model from blocks

  1. Open Build, find Visual objects, and choose a model.
  2. Pick a cube, wide block, tall block, long block, or slab, then pick a color.
  3. Click an empty grid spot in the picture to place it, or select Add piece.
  4. Click any existing piece in the picture or piece tray to select it.
  5. Use the large arrows, Up, Down, Rotate, Copy, Paint, or Erase controls.
  6. Change the active layer with minus and plus. Drag to orbit and use the wheel to zoom.

Every builder action is a normal reversible editor command. Undo, redo, checks, change review, and atomic saving work exactly as they do for advanced fields.

World generation workflow

  • World seed changes only the preview world and is not saved as content.
  • Terrain and feature salts are persisted compatibility data that isolate random streams.
  • Explain passes reports climate, density, cave carving, and the exact block after each pass.
  • Compare saved profile renders edited and generated profiles with the same seed.
  • Verify deterministic generation checks golden coordinates, request order, and pass replay.
  • Selecting one biome forces that biome into the preview and shows its real surface-material palette.
  • Selecting one ore, decoration, or structure isolates its production shape and registered blocks.

The legacy-v1 profile is intentionally read-only. Saving engine-v2 generation changes requires typing MIGRATE because untouched terrain may change for existing seeds.

Entity and asset workflow

  1. Build a procedural model from colored blocks or connect a validated local GLB model.
  2. Edit an entity to connect its model, animation set, faction, behavior, spawn, loot, and interaction data.
  3. Use Spawn selected for one actor or Run spawn rules for deterministic populations.
  4. Use combat and damage controls while reading the live detached blackboard below them.

The isolated test scene uses memory-only persistence. It cannot modify the playable world's local-storage save.

Behavior graph workflow

Graphs are safe typed data rather than executable scripts. Selectors try alternatives, sequences require ordered success, conditions inspect runtime facts, actions produce intent, cooldowns limit frequency, and repeats bound repeated work.

  1. Select a graph and edit node IDs, types, children, conditions, actions, and parameters in the Inspector.
  2. Keep every node reachable from the root and avoid cycles; validation reports the exact broken path.
  3. Run the linked entity, move the test player into range, and inspect target, intent, state, and timers.

Validation, generated code, and saving

The editor validates versions, numeric ranges, IDs, references, asset ownership, graph structure, generation budgets, and compatibility rules across the complete project—not only the selected file.

A successful save checks that the disk revision still matches, stages canonical JSON and generated TypeScript together, and atomically commits the transaction. If another program changed content, reload the editor before saving so those changes are never overwritten.

Do not hand-edit files under src/generated/. They are deterministic compiler output and will be replaced by the next successful generation.
